## 1. 芯片规格

### 1.1 NTAG213 参数

| 参数 | 规格 | 说明 |
|------|------|------|
| 芯片型号 | NXP NTAG213 | NFC Forum Type 2 Tag |
| 工作频率 | 13.56 MHz | ISO 14443A |
| 总内存 | 180 字节 | 含系统保留区 |
| 用户内存 | 144 字节 | 可写入区域 |
| NDEF 可用 | ~137 字节 | 扣除 NDEF 头和 TLV 开销 |
| UID | 7 字节 | 全球唯一，出厂固化，不可修改 |
| 密码保护 | 32-bit PWD + 16-bit PACK | 可设置读/写密码 |
| 读写距离 | 1-4 cm | 需贴近感应 |
| 读写次数 | 10万次 | 足够反复更新 |
| 数据保存 | 10年 | 常温常湿条件 |

### 1.2 内存布局

```
┌──────────────────────────────────────────┐
│  Page 0-1   (8 bytes)   UID 序列号       │  ← 只读，出厂固化
│  Page 2     (4 bytes)   内部/锁定位       │  ← 系统保留
│  Page 3     (4 bytes)   CC (Capability)  │  ← NDEF 容器配置
├──────────────────────────────────────────┤
│  Page 4-39  (144 bytes) 用户数据区        │  ← NDEF 消息写入区
├──────────────────────────────────────────┤
│  Page 40    (4 bytes)   动态锁定位        │  ← 系统保留
│  Page 41    (4 bytes)   CFG0 配置         │  ← 镜像/访问控制
│  Page 42    (4 bytes)   CFG1 配置         │  ← 密码保护起始页
│  Page 43    (4 bytes)   PWD 密码          │  ← 32-bit 密码
│  Page 44    (2 bytes)   PACK             │  ← 密码应答码
└──────────────────────────────────────────┘
```

## 2. 写入内容标准

### 2.1 URL 格式规范

每张 NFC 白卡写入一条 **NDEF URI Record**，URL 格式如下：

```
https://{域名}/h5/#checkin?tag={tag_id}
```

#### 参数说明

| 参数 | 格式 | 说明 | 示例 |
|------|------|------|------|
| 域名 | 生产域名 | 系统访问地址 | `shue.example.com` |
| tag_id | `{类型}_{编号}_{城市}` | NFC标签唯一标识，与数据库 nfc_tags.tag_id 一致 | `house_001_shanghai` |

#### 完整示例

| 用途 | tag_id | 完整 URL |
|------|--------|---------|
| 房源-静安寺公寓 | `house_001_shanghai` | `https://shue.example.com/h5/#checkin?tag=house_001_shanghai` |
| 房源-徐家汇花园 | `house_002_shanghai` | `https://shue.example.com/h5/#checkin?tag=house_002_shanghai` |
| 房源-虹桥路 | `house_003_shanghai` | `https://shue.example.com/h5/#checkin?tag=house_003_shanghai` |
| 门店-总部 | `office_001_main` | `https://shue.example.com/h5/#checkin?tag=office_001_main` |

#### URL 长度估算

```
https://shue.example.com/h5/#checkin?tag=house_001_shanghai
                                                            
协议头 "https://"          →  NDEF 压缩为 1 字节 (前缀码 0x04)
域名+路径+参数             →  约 50-70 字节
─────────────────────────────────────────
总计                       →  约 55-75 字节 (NTAG213 的 137 字节绰绰有余)
```

### 2.2 NDEF 记录结构

写入的 NDEF 消息结构：

```
┌─ NDEF Message ──────────────────────────────────┐
│  TLV Header:                                     │
│    T = 0x03 (NDEF Message)                       │
│    L = 消息长度                                   │
│                                                   │
│  Record 1: URI Record                            │
│    TNF  = 0x01 (NFC Forum well-known type)       │
│    Type = "U" (URI)                              │
│    ID Code = 0x04 (https:// 前缀)                │
│    Payload = "shue.example.com/h5/#checkin?..."   │
│                                                   │
│  Terminator TLV:                                  │
│    T = 0xFE                                       │
└──────────────────────────────────────────────────┘
```

## 3. 写卡操作流程

### 3.1 准备工作

#### 物料清单

| 物料 | 数量 | 说明 |
|------|------|------|
| NTAG213 白卡 | 按房源数量 + 10% 备用 | 建议采购带背胶的贴纸式 |
| Android 手机 | 1台 | 需支持 NFC，Android 5.0+ |
| NFC Tools App | - | Google Play / 应用商店下载 |
| 写卡清单表 | - | Excel：房源名称、tag_id、URL、写卡状态 |

#### 环境要求

- 手机 NFC 功能已开启（设置 → 连接 → NFC → 开启）
- 手机移除厚保护壳（可能影响 NFC 感应距离）
- 准备好写卡清单，按顺序逐张写入

### 3.2 单张写卡步骤（NFC Tools App）

```
第1步: 打开 NFC Tools App
       ↓
第2步: 点击底部 "Write" 标签页
       ↓
第3步: 点击 "Add a record"
       ↓
第4步: 选择 "URL/URI"
       ↓
第5步: 输入完整 URL
       例如: https://shue.example.com/h5/#checkin?tag=house_001_shanghai
       ↓
第6步: 点击 "OK" 确认
       ↓
第7步: 点击 "Write / XX Bytes"
       ↓
第8步: 将手机背面 NFC 感应区贴近白卡
       (通常在手机背部中上方位置)
       ↓
第9步: 听到"滴"声或看到 "Write complete" 提示
       ↓
第10步: 验证 → 切换到 "Read" 标签页 → 再次贴近白卡
        确认读出的 URL 与写入一致
```

### 3.3 批量写卡流程

```
┌─────────────────────────────────────────────────────────────┐
│                     批量写卡流水线                            │
│                                                              │
│  准备阶段:                                                    │
│  ① 导出房源清单 (从运营后台 /admin → 房源管理 → 导出)         │
│  ② 生成写卡清单 (tag_id → URL 映射表)                        │
│  ③ 在白卡上预先用记号笔标注 tag_id 编号                       │
│                                                              │
│  写卡阶段 (建议2人配合):                                       │
│  ④ A读清单报 URL → B在App中输入 → 贴近白卡写入                │
│  ⑤ 写入成功 → 在清单上打勾 ✓                                  │
│  ⑥ 写入失败 → 换一张白卡重试 → 废卡标记 ✗                     │
│                                                              │
│  验证阶段:                                                    │
│  ⑦ 逐张贴近手机，确认自动弹出正确 URL                          │
│  ⑧ 点击 URL 进入 H5 打卡页面，确认 tag_id 正确解析              │
│                                                              │
│  部署阶段:                                                    │
│  ⑨ 将写好的白卡贴在房源入户门内侧 / 门店前台                    │
│  ⑩ 拍照记录贴卡位置，上传至运营后台                            │
│  ⑪ 在后台将 tag_id 与房源绑定 (POST /attendance/nfc-tags)     │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 写卡速度参考

| 方式 | 速度 | 适用场景 |
|------|------|---------|
| 手动单张 (NFC Tools) | ~30秒/张 | 少量补卡、临时新增 |
| 双人流水线 | ~15秒/张 | 首次批量部署 |
| 管理端写卡工具 (Web NFC) | ~10秒/张 | 规模化部署 (需Chrome Android) |

## 4. 密码保护设置（可选）

### 4.1 为什么要设置密码

防止他人用手机覆写标签内容（如改URL指向钓鱼站），保护标签数据完整性。

### 4.2 密码设置方法

使用 NFC Tools Pro（付费版）或 NXP TagWriter：

```
设置 → Password Protection
  PWD:  4字节密码 (例如: 0x53 0x48 0x55 0x45 = "SHUE")
  PACK: 2字节应答 (例如: 0x80 0x80)
  
保护范围: 从 Page 4 开始 (保护整个用户数据区)
保护模式: Write-protect only (允许读，禁止写)
```

### 4.3 密码管理

| 事项 | 规范 |
|------|------|
| 密码统一性 | 同一批次标签使用相同密码 |
| 密码记录 | 密码存储在运营后台系统配置中，不写在纸质文档 |
| 密码更换 | 随标签轮换周期一起更换 |
| 解锁场景 | 需要更新URL或退役标签时解锁 |

## 5. 验证标准

### 5.1 写卡后必须验证

| 验证项 | 方法 | 合格标准 |
|--------|------|---------|
| URL 正确 | NFC Tools 读取 | 读出的URL与写卡清单完全一致 |
| 自动跳转 | 手机靠近白卡 | 弹出URL通知，点击后打开H5打卡页 |
| tag_id 解析 | 在H5页面检查 | 页面正确识别 tag_id 并显示对应房源信息 |
| 密码保护 | 尝试覆写 | 未授权设备无法覆写标签内容 |

### 5.2 常见问题排查

| 问题 | 原因 | 解决 |
|------|------|------|
| 写入失败 | 手机NFC未开启 | 设置→NFC→开启 |
| 写入失败 | 手机壳太厚 | 去掉保护壳重试 |
| 写入失败 | 白卡已被锁定 | 换一张新白卡 |
| 读取无反应 | 感应位置不对 | 调整手机贴近白卡的位置 (背部中上方) |
| URL 乱码 | 编码错误 | 确认URL中无中文字符，纯ASCII |
| 跳转后页面白屏 | 域名未配置 | 确认域名已解析且HTTPS证书有效 |
| tag_id 无法识别 | 后台未绑定 | 在运营后台房源管理中绑定该tag_id |

## 6. 标签部署规范

### 6.1 粘贴位置

| 场景 | 推荐位置 | 原因 |
|------|---------|------|
| 房源 | 入户门内侧，距地面 1.2-1.5m | 进门必经，不易被外人发现和破坏 |
| 门店 | 前台固定位置 或 打卡机旁 | 统一管理，方便上下班打卡 |

### 6.2 粘贴要求

- 表面清洁干燥后再粘贴
- 使用标签自带背胶 + 透明保护膜覆盖
- 避免粘贴在金属表面（会屏蔽NFC信号）
- 标签上不标注任何系统信息（防止社工攻击）

### 6.3 标签编号与登记

每张标签部署后需在系统中登记：

```
POST /attendance/nfc-tags
{
  "tagId": "house_001_shanghai",
  "tagType": "HOUSE",
  "houseId": "house_001",
  "lat": 31.2304,
  "lng": 121.4737,
  "address": "上海市静安区南京西路1266号"
}
```

## 7. 标签生命周期管理

### 7.1 生命周期

```
采购 → 写卡 → 验证 → 部署 → 运行 → 轮换/退役
                                      ↓
                              回收旧卡 → 物理销毁
```

### 7.2 轮换计划

| 阶段 | 时间 | 操作 |
|------|------|------|
| 首次部署 | Q2 2025 | 全量写卡+部署 |
| 第一次轮换 | Q3 2025 (7月) | 新卡写入+更换，旧卡并行2周后失效 |
| 第二次轮换 | Q4 2025 (10月) | 同上 |
| 后续 | 每季度 | 按需轮换损坏/脱落的标签 |

### 7.3 退役处理

- 后台将 tag_id 状态改为 `INACTIVE`
- 物理回收标签并剪碎销毁
- 系统拒绝已退役 tag_id 的打卡请求
