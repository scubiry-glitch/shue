## 1. 需求背景

随着业务复杂度提升，单一的考勤打卡已无法满足不同角色的业务场景需求。需要支持 **多角色 × 多任务** 的精细化打卡体系，为不同业务线提供数据支撑。

### 1.1 核心业务诉求（来自需求研讨会）

> "要实现几个场景。第一个场景就是我们的经纪人，他到房子里边儿一刷就可以确定他来空看过，我就可以判断多少经纪人空看过这套房子。第二个租户管家，他是否在做过房屋检修、做过面对面交付，这两个他要打卡。第三，资管经理有没有去过房屋进行房屋整备。各角色分别打卡，每个人打卡的界面是不一样的，或者他可以下拉菜单自己选。总之，他是多角色多任务的一个确定，最后后台进行统计。"

### 1.2 三大核心场景

| 场景 | 角色 | 核心问题 | 业务价值 |
|------|------|---------|---------|
| **场景一** | 经纪人 | 哪些经纪人空看过这套房？看了几次？ | 衡量房源热度，优化推广策略 |
| **场景二** | 租户管家 | 是否完成了房屋检修？是否完成了面对面交付？ | 确保服务质量，追踪任务完成率 |
| **场景三** | 资管经理 | 是否到现场进行了房屋整备？ | 监督整备进度，保证房源品质 |

## 2. 角色与任务矩阵

### 2.1 角色定义

| 角色编码 | 角色名称 | 所属部门 | 核心职责 |
|----------|----------|----------|----------|
| `AGENT` | 经纪人 | 业务部 | 房源带看、空看、签约 |
| `HOUSE_MANAGER` | 租户管家 | 运营部 | 房屋检修、交付验收、退租检查 |
| `ASSET_MANAGER` | 资管经理 | 资管部 | 房屋整备、装修监督、质量把控 |

### 2.2 任务类型定义

#### 经纪人任务 (AGENT)
| 任务编码 | 任务名称 | 说明 | 是否需签退 | 必须拍照 |
|----------|----------|------|-----------|---------|
| `EMPTY_VIEW` | 空看 | 经纪人独自查看房源，无客户 | 否（一刷即走） | 否 |
| `SHOWING` | 带看 | 带客户看房 | 是 | 是(≥1张) |
| `SIGNING` | 签约 | 现场签约 | 否 | 是(≥2张) |
| `RENEWAL` | 续约谈 | 续租谈判 | 否 | 否 |

#### 租户管家任务 (HOUSE_MANAGER)
| 任务编码 | 任务名称 | 说明 | 是否需签退 | 必须拍照 |
|----------|----------|------|-----------|---------|
| `INSPECTION` | 房屋检修 | 日常检修维护 | 是 | 是(≥3张) |
| `CHECKIN_DELIVERY` | 入住交付 | 面对面交付给租户 | 是 | 是(≥5张) |
| `CHECKOUT_INSPECTION` | 退租验收 | 退租时房屋检查 | 是 | 是(≥5张) |
| `EMERGENCY_REPAIR` | 紧急维修 | 紧急维修处理 | 是 | 是(≥2张) |
| `CLEANING_CHECK` | 保洁验收 | 保洁后验收 | 否 | 是(≥2张) |

#### 资管经理任务 (ASSET_MANAGER)
| 任务编码 | 任务名称 | 说明 | 是否需签退 | 必须拍照 |
|----------|----------|------|-----------|---------|
| `PREPARE_OVERSEE` | 整备监督 | 房屋整备过程监督 | 是 | 是(≥3张) |
| `RENOVATION_CHECK` | 装修验收 | 装修质量验收 | 否 | 是(≥5张) |
| `QUALITY_AUDIT` | 质量巡检 | 定期质量抽查 | 否 | 是(≥2张) |
| `VENDOR_SUPERVISE` | 供应商监督 | 第三方服务监督 | 是 | 否 |

### 3.2 角色识别策略（推荐方案C：混合模式）

```
┌──────────────────────────────────────┐
│ 角色识别逻辑:                         │
│                                       │
│ 1. 根据登录账号，获取用户的主角色       │
│ 2. 打卡页面默认显示主角色的任务列表     │
│ 3. 顶部提供「角色」下拉选择器           │
│    可切换到该用户拥有的其他角色         │
│ 4. 切换角色后，任务列表动态刷新         │
│                                       │
│ 示例:                                  │
│  张三 主角色=经纪人 副角色=租户管家     │
│  → 默认显示: 空看/带看/签约/续约谈     │
│  → 切换管家: 检修/交付/退租/维修/保洁  │
└──────────────────────────────────────┘
```

## 5. 数据模型设计

### 5.1 扩展打卡记录表

```sql
-- 在 attendance_records 表新增字段
ALTER TABLE attendance_records ADD COLUMN role VARCHAR(32);          -- 角色: AGENT/HOUSE_MANAGER/ASSET_MANAGER
ALTER TABLE attendance_records ADD COLUMN task_type VARCHAR(32);     -- 任务类型: EMPTY_VIEW/SHOWING/INSPECTION/...
ALTER TABLE attendance_records ADD COLUMN task_name VARCHAR(64);     -- 任务中文名
ALTER TABLE attendance_records ADD COLUMN require_checkout BOOLEAN DEFAULT FALSE; -- 是否需要签退
ALTER TABLE attendance_records ADD COLUMN task_data JSONB DEFAULT '{}';           -- 任务扩展数据

-- 索引
CREATE INDEX idx_records_role ON attendance_records(role);
CREATE INDEX idx_records_task ON attendance_records(role, task_type);
CREATE INDEX idx_records_house_role ON attendance_records(house_id, role);
```

```
经纪人-空看:     { notes?: string }

租户管家-交付:   { tenantName, tenantPhone, deliveryItems: [{item, status}] }
```

## 6. API 接口设计

### 6.1 获取角色配置

```
GET /api/roles/config
→ 返回所有角色及其任务类型配置
```

### 6.2 多角色打卡

```
POST /api/attendance/checkin
Body 新增: role, taskType, taskData
→ 返回打卡结果 + 是否需要签退提示
```

### 6.3 房源多角色统计

```
GET /api/attendance/house-stats?houseId={id}
→ 返回该房源各角色的打卡统计（空看人次、检修次数、整备进度等）
```

### 6.4 经纪人空看统计

```
GET /api/attendance/stats/empty-views?houseId={id}&period=month
→ 返回该房源空看经纪人明细、人次、趋势
```

### 6.5 租户管家任务完成统计

```
GET /api/attendance/stats/by-role?role=HOUSE_MANAGER&period=month
→ 返回管家任务完成率、各任务类型统计
```
